## Problem Statement

법률/프라이버시 팀이 GDPR 법률의견 작성 시 법령 원문, EDPB 가이드라인, CJEU 판례를 수동으로 크로스레퍼런스하는 데 과도한 시간을 소비하고 있음. PIPA-expert(한국 개인정보보호법 전문 에이전트)와 동일한 아키텍처로 EU 데이터 보호 법령 전문 에이전트를 구축하여, 법령 원문 + 공식 가이드라인 기반의 신뢰할 수 있는 RAG 시스템을 제공한다.

## Demand Evidence

- PIPA-expert가 이미 성공적으로 구축 완료 (10개 법률, 929개 조문, 46개 PIPC 가이드라인)
- 동일한 법률팀이 EU GDPR 관련 업무에서도 같은 pain point을 경험
- 현재 48개 파일의 knowledge base가 이미 수작업으로 구축됨 — 니즈가 입증된 상태

## Status Quo

현재 GDPR 법률 리서치 워크플로우:
- EUR-Lex 웹사이트에서 GDPR 원문 검색 (Ctrl+F)
- EDPB 웹사이트에서 가이드라인 PDF 열어서 수동 검색
- GDPRhub 등에서 판례 개별 검색
- 이 모든 소스를 크로스레퍼런스하여 의견서 작성

## Target User & Scope

**대상**: 사내 법률/프라이버시 팀 — GDPR 관련 법률의견, 컴플라이언스 리뷰, DPIA 작성을 담당하는 변호사/DPO

**법령 범위** (5개):

| 법령 | CELEX | 조문 수 (추정) | 우선순위 |
|------|-------|-------------|---------|
| GDPR (Regulation 2016/679) | 32016R0679 | ~99 Articles + 173 Recitals | P0 |
| ePrivacy Directive (2002/58/EC) | 32002L0058 | ~19 Articles | P0 |
| EU AI Act (Regulation 2024/1689) | 32024R1689 | ~113 Articles | P1 |
| Data Act (Regulation 2023/2854) | 32023R2854 | ~50 Articles | P1 |
| Data Governance Act (2022/868) | 32022R0868 | ~44 Articles | P1 |

**보조 자료 범위**:

| 카테고리 | 초기 목표 | 최종 목표 | 소스 |
|---------|----------|----------|------|
| EDPB Guidelines | 82+ (전수) | 82+ | edpb.europa.eu |

## Constraints

1. **PIPA-expert와 구조적 일관성 유지** — 동일한 directory layout, frontmatter schema 기반 + EU-specific 확장 (아래 상세), source grading 체계
2. **EUR-Lex API 사용** — EUR-Lex SOAP API 확보 완료. 공식 API로 법령 수집 (Formex XML → 조문별 분리)
3. **영어 기본** — EU 법령은 영어 버전 기본 수집 (한국어 번역은 향후)
4. **기존 48개 파일 보존** — 리팩토링하되 콘텐츠 삭제 없음. 기존 monolithic GDPR 원문은 `library/inbox/archive/`로 보존
5. **v1.0 RAG은 키워드 인덱스 + Grep 기반** — 벡터 검색은 v2.0
6. **최신 통합본(consolidated version) 수집** — EUR-Lex에서 개정 반영된 최신 통합본 사용
7. **P1 법령(AI Act, Data Act, DGA)은 수집 우선, 에이전트 통합은 후순위** — GDPR + ePrivacy가 에이전트에 완전 통합된 후 P1 법령 추가

## Premises

1. PIPA-expert 아키텍처가 EU 법령에도 적용 가능하다. 단, EU 법령 고유 구조(Recitals, Chapter/Section 계층, `Article X(Y)(Z) point (a)` 형식 cross-reference)에 대한 적응이 필요함
2. EUR-Lex SOAP API 확보 완료 — Formex 4 XML 포맷으로 법령 다운로드 가능
3. EDPB 가이드라인 82개를 PDF→md 파이프라인으로 처리 가능하다. 단, 테이블/차트/각주가 있는 PDF는 변환 품질 QA 필요
4. 5개 법령의 조문 총합 (~325개 + 173 Recitals)은 관리 가능한 규모이다
5. 현재 48개 파일의 콘텐츠 품질이 유지할 가치가 있다

### EU vs Korean Law — 구조적 차이점

PIPA-expert 아키텍처를 EU에 적용할 때 고려해야 할 차이:

| 차이점 | Korean (PIPA) | EU (GDPR) | 적응 방법 |
|-------|---------------|-----------|----------|
| **해석 보조 자료** | 없음 | Recitals (전문) — 별도 해석 레이어 | `gdpr-recitals/` 별도 디렉토리, cross-ref로 연결 |
| **조문 계층** | 조 → 항 → 호 → 목 | Chapter → Section → Article → Paragraph → Point → Sub-point | `_hierarchy.json`에 Chapter/Section 계층 추가 |
| **Cross-reference 포맷** | 제15조제1항제1호 | Article 6(1)(a) point (i) | EU 전용 파서 필요 — 기본 regex: `Article\s+\d+(?:\(\d+\))*(?:,?\s*(?:first\|second\|third)?\s*(?:sub)?paragraph)?(?:,?\s*point\s*\([a-z]\))*`. GDPR 특수 패턴(subparagraph, indent) 추가 처리 필요 — Phase 1에서 실제 corpus 기반으로 반복 개선 |
| **Sub-article 번호** | 조의2, 조의3 (article_sub) | 없음 | `article_sub` 필드는 항상 0, 유지하되 사용하지 않음 |
| **법령 간 참조** | 동일 법 체계 내 위임 | Regulation ↔ Directive 간 참조 + Recital 참조 | `cross_references`에 법령 식별자 포함: `"ePrivacy Art. 5(3)"` |

### EUR-Lex Collection Script Design

**`scripts/fetch-eu-legislation.py`:**

```
입력: CELEX 번호 리스트 (e.g., ["32016R0679", "32002L0058", ...])
출력: library/grade-a/{law-dir}/art{N}.md + JSON 메타데이터

1. EUR-Lex SOAP API `doQuery` 호출 → CELEX 번호로 법령 검색
2. `getDocument` 호출 → Formex 4 XML 다운로드
   - Formex XML 구조: <CONS.ACT> → <TITLE> → <PREAMBLE> (Recitals) → <ENACTING.TERMS> (Articles)
   - Articles: <ARTICLE ID="art_6"> → <TI> (제목) → <ALINEA>/<LIST> (내용)
   - Recitals: <GR.CONSID> → <CONSID> (개별 Recital)
3. XML 파싱 → Chapter/Section/Article 계층 추출
4. 각 Article에 대해:
   - YAML frontmatter 생성 (law, law_id, article, chapter, title, cross_references, keywords)
   - Cross-reference 파싱: regex `Article\s+\d+(\(\d+\))*` 패턴으로 추출
   - Markdown body 생성 (조문 원문, paragraph 번호 보존)
   - library/grade-a/{law-dir}/art{N}.md 로 저장
5. Recitals 별도 처리 → library/grade-a/{law-dir}-recitals/recital{N}.md
6. _meta.json, _hierarchy.json (Chapter→Section→Article 트리), _cross-refs.json 생성
7. article-index.json 업데이트

에러 처리: API 호출 실패 시 exponential backoff (1s, 2s, 4s, max 3회)
로깅: 수집 진행률 + 실패 항목을 scripts/logs/{datetime}.log에 기록
```

**API 전략**: EUR-Lex SOAP API를 primary로 사용 (Formex XML 구조화된 출력).
CELLAR REST/SPARQL은 SOAP API 장애 시 fallback 또는 메타데이터 보강용.

**CELEX → CELLAR UUID 매핑** (fallback용):
```sparql
# SPARQL endpoint: https://publications.europa.eu/webapi/rdf/sparql
SELECT ?work ?expression WHERE {
  ?work cdm:resource_legal_id_celex "32016R0679" .
  ?expression cdm:expression_belongs_to_work ?work .
  ?expression cdm:expression_uses_language <http://publications.europa.eu/resource/authority/language/ENG> .
}
```

**Formex XML 파싱 참고**: Formex 4는 default namespace 없이 태그명만 사용 (`<CONS.ACT>`, `<ARTICLE>` 등). `lxml`에서 namespace-unaware 파싱 가능. 상세 스키마: https://op.europa.eu/en/web/eu-vocabularies/formex

### Agent Design (PIPA-expert 미러)

**`gdpr-agent.md`** — GDPR/EU 데이터 보호법 전문가 페르소나

1. **Search Protocol (4-step):**
   - Step 1: `article-index.json` → 키워드 매칭 (조문 제목/키워드)
   - Step 2: `guideline-index.json` → EDPB 가이드라인 매칭
   - Step 3: Cross-reference 추적 → 관련 조문/Recital 읽기
   - Step 4: Web search fallback (EUR-Lex, EDPB, 로펌, 학술)

2. **Citation System:**
   - `[VERIFIED] [Grade A]` — 법령 원문/EDPB 가이드라인
   - `[VERIFIED] [Grade B]` — CJEU 판례/집행결정
   - `[UNVERIFIED]` — KB에 없는 소스
   - `[CONTRADICTED]` — 소스 간 충돌

3. **Fact-Checker Sub-agent:**
   - 조문 존재 확인, 인용 정확성 검증
   - GDPR 조문 번호 정밀도 (Art.6 vs Art.6(1)(a) 구분)
   - 가이드라인 인용 검증
   - 신뢰도 스코어: 90%+ PASS, 70-89% WARN, <70% FAIL

## What I noticed about how you think

- 이미 PIPA-expert를 성공적으로 구축한 경험이 있어서, "무엇을 만들지"가 아니라 "어떻게 확장할지"에 집중하고 있다. 이건 실행력이 있다는 신호.
- DSA/DMA를 넣을지 스스로 질문한 것 — 스코프 감각이 좋다. "다 넣자"가 아니라 "이건 오바인가?"를 먼저 물었다.
- Data Act/DGA를 추가한 선택은 "개인정보보호"에서 "데이터 거버넌스"로 자연스럽게 확장하는 것 — 법령 간 상호참조 가치가 높은 영역.
- 풀 패리티를 선택한 것 — 검증된 아키텍처를 재활용하겠다는 판단. 불필요한 실험을 피하는 실용적 접근.
